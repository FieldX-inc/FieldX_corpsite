# SEO記事パイプライン

## 概要

この skill は司令塔として振る舞う。  
自分で全部を書くのではなく、必要に応じて既存 skill を順番に使い、1本の記事を publish-ready に近い状態まで持っていく。  
specialist skill は必要なら sub-agent として起動してよいが、常時すべて委譲しない。最終的な企画判断、ブランド判断、CTA の最終決定、公開可否判定はこの orchestrator が持つ。

使う specialist skill:

- `$keyword-strategy`
- `$serp-analysis`
- `$content-outline`
- `$seo-writer`
- `$conversion-cta`
- `$quality-audit`

## 受け取る入力

最低限ほしい入力:

- テーマまたは主題
- 想定読者
- 事業上の目的
- 想定 CV

あると精度が上がる入力:

- 既存記事との関係
- 避けたい表現
- 参照すべき社内情報
- publish 期限

入力が不足していても止まらず、`AGENTS.md`、`briefs/`、`references/`、`content/` から補完する。

## 事前前提

- `/column` の公開ソースは `microCMS` とみなす。
- 企画開始前に、`MICROCMS_SERVICE_DOMAIN`、`MICROCMS_API_KEY`、`MICROCMS_COLUMN_ENDPOINT` を前提として、公開済み記事の最新状態を確認する。
- 公開記事確認ができない場合、その企画は確定案として扱わず、`公開記事確認未実施` の暫定企画として返す。
- 最優先の事業目的は、AI 活用を検討する賃貸管理会社のリード獲得とする。

## 共通入力契約

下流 skill に渡す前に、少なくとも次を揃える。

- `公開記事スナップショット`
- `記事化判定` (`新規` / `改稿推奨` / `見送り`)
- `内部リンク候補`

`公開記事スナップショット` は次の項目を必須とする。

- `id`
- `title`
- `publishedAt`
- `主要テーマの要約`
- `想定読者`
- `内部リンク候補として使える論点`
- `カニバリ懸念の有無`

## 基本フロー

1. テーマ確認
2. 公開記事確認
3. 企画判定
4. キーワード戦略
5. SERP 分析
6. brief 作成
7. 構成設計
8. 本文執筆
9. CTA 設計
10. 品質監査
11. 最終出力の整理

## 各ステップ

### 1. テーマ確認

- 記事の主題を 1 文で定義する。
- 読者が記事を読む理由を 1 文で定義する。
- 記事読後に取ってほしい行動を 1 つ決める。

ここが曖昧なら、その後の工程も曖昧になる。先に固定する。

### 2. 公開記事確認

- `microCMS` の公開済み記事一覧を取得する。
- 企画候補に近い記事を拾い、`公開記事スナップショット` に標準化する。
- 公開記事ごとに、テーマ、想定読者、使えそうな内部リンク論点、カニバリ懸念を整理する。
- この確認は repo 内の過去ファイルでは代替しない。企画可否の primary source は公開済み `microCMS` 記事とする。

### 3. 企画判定

- 強い重複があれば、新規記事ではなく改稿または統合案を優先する。
- 重複しない場合でも、既存公開記事へ送れる内部リンクを最低 2 件探す。
- 最終目的である賃貸管理会社リード獲得との接続が弱いテーマは、`見送り` 候補として扱う。
- `記事化判定` は必ず `新規` / `改稿推奨` / `見送り` のいずれかで明示する。

### 4. キーワード戦略

`$keyword-strategy` を使って、主軸キーワードと補助キーワードを決める。  
記事化を見送るべきなら、ここで止めてもよい。

### 5. SERP 分析

`$serp-analysis` を使って、現在の検索意図と差別化の勝ち筋を整理する。  
最新情報が必要な論点はこの段階でフラグを立てる。

### 6. brief 作成

`templates/article-brief.md` に沿って brief をまとめる。  
保存先は `briefs/` とし、ファイル名は `YYYY-MM-DD-topic.md` を基本にする。

brief に必ず含める:

- 主軸キーワード
- 読者像
- 検索意図
- 記事の勝ち筋
- 必須論点
- 避ける論点
- CTA 方針
- 検証が必要な主張
- 企画前に確認した公開済み記事一覧の要約
- カニバリ判定
- 新規記事で行く理由 または 改稿提案
- 内部回遊設計メモ

### 7. 構成設計

`$content-outline` を使って、brief から構成案を作る。  
出力は `templates/outline-template.md` に沿わせる。

### 8. 本文執筆

`$seo-writer` を使って、構成案から本文を作る。  
このとき以下も一緒に作る。

- タイトル案
- meta description 案
- FAQ 案
- 図解が必要な箇所

### 9. CTA 設計

`$conversion-cta` を使って、記事の温度感に合う導線を設計する。  
CTA は本文を書き終えてから後付けするのではなく、記事の結論と矛盾しないように整える。

### 10. 品質監査

`$quality-audit` を使って公開判定を行う。  
重大な問題があれば修正し、必要なら再監査する。

### 11. 最終出力の整理

最低限まとめるもの:

- 企画前に確認した公開済み記事一覧の要約
- カニバリ判定
- 新規記事で行く理由 または 改稿提案
- brief
- 構成案
- 本文ドラフト
- title / description 候補
- CTA
- QA 結果
- 内部回遊設計メモ

## sub-agent を使ってよい場面

sub-agent は並列で効果が出る調査だけに使う。  
specialist skill を毎回全部 sub-agent 化するのではなく、並列化の価値が高い工程だけ委譲する。

使ってよい例:

- 公開記事一覧の要約整理
- 上位競合ページの要約収集
- 関連質問の抽出
- 一次情報ソース候補の収集
- repo 内の内部リンク候補探索

使わない例:

- 最終的な記事方針の決定
- ブランド表現の判断
- CTA の最終決定
- 公開可否の最終判断

最終統合は必ず自分で行う。

## 出力形式

最後は次の順で返す。

1. 企画前に確認した公開済み記事一覧の要約
2. カニバリ判定
3. 新規記事で行く理由 または 改稿提案
4. 採用キーワード
5. brief の要点
6. 構成
7. 本文ドラフト
8. CTA
9. 内部回遊設計メモ
10. QA 結果
11. 未解決の確認事項

## 運用ルール

- 公開ソースが `microCMS` であることを前提にする。
- 生成物はそのまま publish 済みとみなさない。
- 競合比較、最新統計、法規制、価格、制度、プロダクト仕様は検証前提とする。
- 日本語のみで出力する。
- 既存記事との cannibalization が強い場合は、新規記事ではなく改稿を提案してよい。
- 公開記事確認に失敗した場合は、その不足を明示し、暫定企画として扱う。
